3주간 반복적으로 마주친 **본질적 문제 7가지**. 각각에 대응하는 처방을 만들고 패턴화했다.

## 3. 트러블슈팅 대표 사례

### 사례 1: Toast XSS 위험 — Wave 1.5 reviewer catch (CRITICAL)
- **상황**: Toast 메시지에 `innerHTML` 사용 → 사용자 입력 섞이면 XSS 가능성
- **catch**: reviewer가 1차 리뷰에서 발견
- **수정**: `textContent` + DOM 조립 패턴으로 전환
- **교훈**: 단독 작성자였으면 보안 결함이 그대로 머지됐을 가능성

### 사례 2: recordSpecialty dedup gate 누락 — Wave 2 #4 reviewer catch (CRITICAL)
- **상황**: 같은 아이템 ID가 두 번 호출되면 특산품 카운트가 두 번 증가 → 다음 마을이 의도보다 빨리 해제될 수 있음
- **catch**: reviewer가 호출 흐름 검증 중 발견
- **수정**: ProgressSystem에 dedup 가드 추가
- **교훈**: 호출 sequence 결함은 단위 테스트만으론 catch 어려움

### 사례 3: 빗살무늬 디버깅 — Wave 3 #1 (5/5)
- **상황**: 고양이 색을 밝게 바꾸면 표면에 줄무늬가 보임
- **분석**: 텍스처 추출 → 픽셀 휘도 분포 분석 → 4가지 보정안 시뮬레이션 PNG 생성
- **결정**: 사용자가 시뮬 4종 비교 → "원래 방식이 가장 자연스럽다" 결론 → 롤백
- **교훈**: 시각 검증은 결국 사람의 눈. AI가 데이터로 후보를 가지런히 정렬하면 결정은 빠름

### 사례 4: 모달 닫기 시 게임 입력 영구 잠김 — Wave 3 #2 reviewer catch (CRITICAL)
- **상황**: ESC/취소/✕/배경 클릭 시 `setInputEnabled(true)`가 호출 안 돼 게임이 새로고침해야 동작
- **catch**: reviewer 1차 리뷰에서 즉시 발견
- **수정**: `onClose` 콜백을 모든 닫기 경로에서 호출
- **교훈**: 통합 결함(여러 코드 경로 교차) catch에 reviewer 분리가 명확히 효과

### 사례 5: 게이트 메커니즘 4단계 진화 — Wave 3 #2 playtest (5/6~8)
- **상황**: 캡 메커니즘 도입 후 3차례 회귀 발견
  - (a) 새 마을 진입 시 게이지가 안 풀림
  - (b) 특산품 2/3에서 미리 풀림
  - (c) 뒷마을로 돌아가면 캡이 풀림
- **수정 흐름**: 매번 사용자 플레이 → 보고 → 수정 + 단위 테스트 추가
- **종착**: "최상위 unlocked region 단일 게이트" 모델로 수렴
- **교훈**: 게임 디자인은 사전에 다 못 정함. 짧은 루프 + 회귀 가드 패턴이 안전망

### 사례 6: 자율 down-grade의 실재 — 전 기간
- **상황**: 사용자가 무거운 워크플로우를 명시 호출했지만 에이전트가 더 가벼운 경로로 자율 전환한 사례 3건
  - `/deep-interview` 호출 × 2회 → 작은 결정엔 과잉으로 판단, 직접 실행으로 down-grade
  - "OMC team/autopilot 사용 여부" → 단독 Level 2가 더 적합으로 판단해 자율 거절
- **결과**: 3건 모두 사용자 사후 동의로 종결, 결과물 수용
- **교훈**: 양날의 검 — 자동 down-grade가 사용자 의도를 잘못 읽으면 신뢰 붕괴 위험. 거절 사유를 매번 명시 + 재확인 패턴 정착 필요

### 사례 7: Z-fighting 근본 수정 — Phase 2~4 (4/20)
- **상황**: 같은 평면에 그려진 ground와 occlusion mesh가 깜빡거림
- **시도**: 처음엔 z-offset으로 땜질 → 다른 chunk 경계에서 재발
- **종착**: shared-material occlusion transfer 패턴으로 근본 해결
- **교훈**: 시각 결함은 땜질로 가리지 말고 root cause를 찾자

## 4. 느낀 점 (인사이트 누적)

### ✅ 1. Phase 0 사전 합의 비용 vs 효과
- ralplan-DR consensus는 Phase 0에서 **CRITICAL 5 + MAJOR 3을 코드 작성 0줄 시점에 사전 적발**
- 코드 작성 후 발견했다면 reviewer 5~10세션이 필요했을 추정
- **사전 합의 단계 투자가 유리한 결정 규모를 가늠하는 가설로 기록**

### ✅ 2. 자가 승인 루프는 Level 2가 차단
- Level 2 적용 작업에서 발견된 CRITICAL **8건**, HIGH **9건**은 모두 reviewer 단계 catch
- Level 1만 사용한 단계의 같은 결함은 사후 hotfix(Wave 1·1.5 hotfix 4건)로 잡힘
- **분리(작성↔검토)의 구조적 가치 = 자가 승인 차단**

### 💾 3. 리뷰 문서 외장기억 — 4일 공백 후 즉시 복귀
- 04-24 → 04-28 4일 공백 뒤 재진입 시, 21개 review 문서로 **컨텍스트 즉시 복구**
- "지금까지 뭐했지" 재탐색에 토큰·시간 소비 거의 없음
- 사람-AI 협업의 본질적 약점(세션 휘발) 보완

### 🎯 4. deep-interview는 큰 결정 1회에만 ROI
- 카피 수정·micro polish에서 재호출 시도 있었으나 5라운드 인터뷰는 과잉
- → 직접 실행으로 down-grade
- **메모리에 "Level 자동 판단" 휴리스틱 등록** → 이후 자동 적용

### ⚠️ 5. DOM/실기 영역은 자동 테스트로 닫히지 않음
- Wave 1.5 hotfix 2건 모두 사람 눈으로 발견된 UX 결함 (WASD+Shift 동시 입력 어려움, 사운드 권장 누락)
- Wave 3 빗살무늬도 같은 패턴
- **매뉴얼 체크리스트 정착 필요성 재확인**

### 🔄 6. 에이전트의 자율 down-grade가 실제로 작동
- 3건 모두 사용자 사후 동의로 종결
- 다만 거절 사유 명시 + 재확인 패턴이 필수

### 🐱 7. 멀티 에이전트는 지금 단계엔 아직 부분적 (Wave 3 발견)
- 게임 디자인 가설은 플레이로 발견됨 → 사전 spec freeze 불가능 → 자율 루프 발산 위험
- **HITL이 부정적 기본값이 아니라, discovery 단계의 적정 도구**라고 인지
- 다음 phase 트리거(Wave 4 대형 시스템 / Playwright MCP 도입)를 명문화

## 🧭 비개발 팀원을 위한 용어 1줄 정리

- **Phase 0** = 코드를 한 줄도 안 쓰고 합의부터 보는 사전 단계
- **Wave** = 굵직한 기능 묶음 (Wave 1 = 퀵윈, Wave 2 = 지역 특산품, Wave 3 = 커스터마이징)
- **HITL** = Human-in-the-Loop. 사람이 중간중간 끼는 작업 방식
- **에이전트** = 특정 역할(작성/검토/계획 등)에 특화된 AI 인스턴스
- **ralph** = AI가 스스로 끝까지 가는 자율 루프 모드
- **Level 1·2·3** = 직접 실행 / 작성+검토 분리 / 자율 루프
- **CRITICAL/HIGH/MEDIUM** = 검토자가 적발한 결함의 심각도 등급
- **회귀(regression)** = 한 곳을 고치면서 다른 곳이 망가지는 현상
- **토큰** = AI가 글자를 처리하는 단위(요금 산정 기준)
- **게이트(gate)** = 다음 단계로 넘어가는 조건
