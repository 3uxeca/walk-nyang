## 1. 최종 목표

> "기획→설계→구현→리뷰→문서→배포까지의 풀 사이클을 사람의 손 거의 안 빌리고 에이전트가 돌릴 수 있는가, 어디까지 닿는가."

walk3d는 그 검증 케이스. 한 사이드 결과물(작동하는 웹게임)이 나오면서 동시에 워크플로우의 한계·이득이 어디서 갈리는지를 데이터로 남긴다.

### 4-3. 정성 인사이트 (관찰 기반, 통제 실험 아님)

1. **자가 승인 루프는 Level 2가 차단했다.** Level 2를 적용한 5건에서 발견된 CRITICAL 2건과 HIGH 4건은 모두 reviewer 단계에서 발견됐다. Level 1만 사용한 단계에서는 이 같은 결함이 사후 hotfix로 잡혔다 (Wave 1 hotfix 2건 + Wave 1.5 hotfix 2건). **샘플은 작지만 패턴이 일관됨.**
2. **deep-interview는 큰 결정 1회에만 ROI.** 카피 수정·micro polish에서 재호출 시도가 있었지만 5라운드 인터뷰는 과잉으로 판단되어 직접 실행으로 전환. 메모리에 "Level 자동 판단" 휴리스틱 등록.
3. **리뷰 문서 외장기억 효과를 4일 공백 후 확인.** 04-24 → 04-28 4일 공백 뒤 재진입 시, 17개 review 문서로 컨텍스트 즉시 복구. 평소라면 "지금까지 뭐했지" 재탐색에 토큰·시간 소비.
4. **DOM/실기 영역은 자동 테스트로 닫히지 않았다.** Wave 1.5 hotfix 2건은 모두 사람 눈으로 발견된 UX 결함(WASD+Shift 동시 입력 어려움, 사운드 켜기 권장 누락)이었다. 매뉴얼 체크리스트 정착 필요성 재확인.
5. **컨센서스 비용 vs 효과의 비대칭.** ralplan consensus는 Phase 0에서 CRITICAL 5 + MAJOR 3을 사전 적발 (코드 작성 0줄 시점). 코드 작성 후 발견했다면 같은 수의 결함을 잡는 데 reviewer 5–10세션이 필요했을 추정. **사전 합의 단계 투자가 유리한 결정 규모를 가늠하는 가설로 기록.**
6. **에이전트의 자율적 down-grade가 실제로 작동했다 (P6 발화 증거).** 사용자가 무거운 워크플로우를 명시 호출했지만 에이전트가 스코프 판단상 더 가벼운 경로로 전환한 사례 3건:
   - `/deep-interview` 호출 × 2회 (사운드 카피·튜토리얼 UI 결정 / 안 쓰이는 건물 조사·매트릭스 브레인스토밍) → "Socratic 5라운드는 과잉" 사유로 직접 조언 또는 Explore 에이전트로 down-grade
   - "OMC team/autopilot 사용 여부" → 에이전트가 먼저 "단독 Level 2가 설계 감각·리뷰 품질에 유리"로 판단해 자율 거절
   - 3건 모두 **사용자 사후 동의로 종결**, 결과물 수용. 단, 이는 양날의 검 — **자동 down-grade가 사용자 의도를 잘못 읽으면 워크플로우 신뢰가 깨질 위험**이 있어, 거절 사유를 매번 명시 + 재확인하는 패턴을 동시에 유지 중.

- writer 1 패스 (Level 2 Wave 단위) = ~10–20k
- 17 리뷰 라운드 합계 = 약 **500k–1M tokens**

## 7. 알려진 한계 / 솔직한 미해결

- **Level 3 자율 루프 검증 불충분.** Phase 1 v1 1회만 사용했고 그 이후 Level 1·2만 사용. E1으로 데이터 추가 예정.
- **A/B 통제 없음.** Level 1 vs Level 2 비교는 시간 순 관찰일 뿐. 같은 작업을 두 방식으로 돌려본 적 없음.
- **토큰 회계 정확도 낮음.** `/cost` 캡쳐 시작 전이라 추정치만 가능. E3에서 보강.
- **에이전트 간 토큰 분배 미측정.** code-reviewer 1회가 writer 1회 대비 얼마나 더 비싼지 정량화 못 함.
- **자동 테스트가 못 잡는 영역의 회귀 보호 인력 비용은 환산 안 됨.** 매뉴얼 체크리스트 실행에 사람 시간이 들어가지만 추적 안 함.
