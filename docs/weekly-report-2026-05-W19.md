## 1. 무엇을 만들었나

### ① 고양이 색상 바꾸기
- 8가지 프리셋(오렌지/화이트/코코아/그레이/블랙/크림/핑크/민트) 중 선택 → 즉시 반영 + 자동 저장
- 모바일·데스크탑 모두 동작
- 기본 색은 블랙으로 변경(요청 반영)

### ② 마을 진행도 정보 패널 + 게이트 메커니즘
- 화면 좌상단 HUD에 **고양이 이름**(기본 `산책냥`, 편집 가능) + **특산품 진행도** `n/3` 표시
- 게임 흐름 정리:
  - **일반 아이템(별·코인·보석)** = 캐릭터 레벨 게이지
  - **특산품(꽃·물고기·네잎클로버·물방울)** = 다음 마을 해제 조건 (지역마다 3개)
  - 특산품을 다 모으기 전엔 일반 아이템 게이지가 **자동으로 캡** → "특산품 N개 더 모아서 다음 마을 가자!" 토스트 안내
  - 캡 상태에선 화면에 흩어진 일반 아이템이 **자동 정리** → "이거 더 줍는다고 진행 안 되네" 혼란 제거
  - 잠긴 마을에 들어가려 하면 "🌸을 N개 더 모으면 잠금 해제!" 친절 안내

## 2. 만들면서 만난 문제

| # | 어디서 | 문제 | 어떻게 알았나 |
|---|--------|------|---------------|
| 1 | 색상 변경 시 | 고양이 색을 밝게 하면 표면에 빗살 같은 줄무늬가 보임 | 사용자가 게임 화면에서 발견 |
| 2 | 정보 패널 모달 | 닫기/취소/ESC 시 게임 입력이 **영원히 잠겨버림** (새로고침해야 풀림) | AI 검토자(reviewer) 에이전트가 catch |
| 3 | 게임 디자인 인지 | "일반 아이템 110/120 채워도 다음 마을 안 풀리는데?" | 사용자가 플레이하다 의문 제기 |
| 4 | 캡 메커니즘 추가 후 회귀 3연발 | (a) 새 마을 가도 게이지 안 풀림 (b) 특산품 2/3에서 미리 풀림 (c) 뒷마을 돌아가면 회귀 | 매번 사용자가 플레이 → 발견 → 보고 |

> 4번이 가장 흥미롭습니다. **한 번에 다 못 보고 플레이마다 새 이슈가 드러난** 패턴.

## 3. 트러블슈팅 경험

### 시각 버그 (#1) — 데이터로 가설 검증
- 게임 텍스처 파일을 추출 → 픽셀 밝기 분포 분석 → 4가지 보정안을 PNG로 시뮬레이션
- 사용자가 시뮬 이미지 4종을 비교 → "원래 방식이 가장 자연스러워" 결정 → 롤백
- **교훈**: AI가 데이터 시각화로 후보를 미리 보여주면 의사결정이 빨라짐

### 모달 입력 잠김 (#2) — AI 검토자의 가치
- 작성 에이전트가 만든 코드를 별도 검토 에이전트가 1차 리뷰
- 검토자: "ESC/취소/저장 등 **모든 닫기 경로**에서 입력 게이트 풀어주는 콜백이 빠졌습니다 — CRITICAL"
- 작성자 단독이라면 놓쳤을 가능성 큰 통합 결함
- **교훈**: 분리(작성↔검토) 워크플로우의 본질적 가치 = "자가 승인 루프 차단"

### 게이트 모델 4단계 진화 (#3, #4)
- 플레이 → 발견 → 수정 → 새 회귀 → 다시 수정을 5회 반복한 끝에
- "**최상위 해제된 마을**의 특산품 진행을 기준으로 게이트를 잡는다"는 단일 모델로 수렴
- 각 수정 단계마다 **단위 테스트 추가**해 같은 회귀 재발 차단 (177 tests passed, 0 회귀)
- **교훈**: 게임 디자인은 사전에 다 못 정함. 짧은 루프 + 회귀 가드 패턴이 안전망

## 4. 느낀 점

### 핵심 인사이트
- **AI 멀티 에이전트(완전 자율 모드)는 지금 단계에 잘 안 맞음**
  - 이유: 게임은 직접 해봐야 디자인 문제가 발견됨 → 사전 스펙을 다 못 적음 → 자율 루프가 발산함
  - 결론: 사람-AI 짧은 루프(HITL, Human-in-the-Loop)가 가장 빠름
- **다만 "작성 ↔ 검토" 분리는 단독보다 명확히 효과적**
  - Wave 3 #2에서 CRITICAL 1건을 1차 검토에서 catch → Level 1 단독이라면 사용자 새로고침 후 발견했을 패턴
- **자율 루프(ralph)는 "스펙이 미리 굳어진" 콘텐츠성 작업에 적합**
  - Wave 3 #1 색상 UI는 ralph 7라운드로 거의 자동 처리
  - Wave 3 #2 게임플레이 튜닝은 부적합 — playtest 피드백 6건이 spec에 없던 결정

### 작업 방식별 적합도 (가설)

| 작업 성격 | 권장 | 이유 |
|-----------|------|------|
| 콘텐츠성 (UI 프리셋, 메뉴 추가 등) | **Level 3 (자율 루프 ralph)** | 스펙이 명확, 검증 자동화 가능 |
| 게임플레이 튜닝 (밸런스, 게이트, 토스트) | **Level 2 (작성자+검토자)** | 통합 결함 catch, 사용자 개입 다수 |
| 1~2줄 폴리싱 (기본값 변경 등) | **Level 1 (직접 실행)** | 에이전트 오버헤드 > 작업 시간 |

## 🧭 비개발 팀원을 위한 용어 1줄 정리

- **HITL** = Human-in-the-Loop. 사람이 중간중간 끼는 작업 방식
- **에이전트** = 특정 역할(작성/검토/계획 등)에 특화된 AI 인스턴스
- **ralph** = AI가 스스로 판단하며 끝까지 가는 자율 루프 모드
- **CRITICAL/HIGH/MEDIUM** = 검토자가 적발한 결함의 심각도 등급
- **회귀(regression)** = 한 곳을 고치면서 다른 곳이 망가지는 현상
- **토큰** = AI가 글자를 처리하는 단위(요금 산정 기준)
- **게이트(gate)** = 다음 단계로 넘어가는 조건
